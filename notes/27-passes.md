# What "passes" are in rustc

- **Upstream commit:** `4f8ea98e`
- **Area:** `compiler/rustc_mir_transform/src/pass_manager.rs`, `compiler/rustc_passes`, `compiler/rustc_lint`
- **Date:** 2026-06-15

## Question

What does "pass" mean in rustc, given that the compiler is described as query-driven rather than running a fixed sequence of passes?

## Short answer

rustc has no single global "list of passes" executed top to bottom; most of the compiler is driven on demand by the query system. The word "pass" survives in a few concrete senses: (1) **MIR passes** — types implementing `MirPass`, run as ordered lists by the `pass_manager` (`run_passes`) with explicit `MirPhase` transitions; (2) the **`rustc_passes` crate** — a grab-bag of HIR-level analysis checks (entry point, lang items, reachability, dead code, stability), themselves exposed *as queries*; and (3) **lint passes** — the `EarlyLintPass`/`LateLintPass` traits. So "pass" names a unit of transformation/checking; "query" names how it's scheduled and memoized.

## Sense 1: MIR passes (`MirPass` + `pass_manager`)

The most literal use. A MIR pass implements `MirPass<'tcx>` (`pass_manager.rs:85`): `name` (`:86`), `run_pass(&self, tcx, body)` mutating one `Body` (`:105`), and two gates — `is_enabled(sess)` (`:95`) and `is_required()` (pure opts return `false`; can be disabled by `#[optimize(none)]`, `:111`). Adapters: `Lint<T>` wraps a read-only `MirLint` (`:119`), `WithMinOptLevel<T>` enables a pass only at/above a `mir_opt_level` (`:160`, via the `o1(...)` helper, `lib.rs:701`).

**How a list runs.** `run_passes` (`pass_manager.rs:204`) takes an ordered `&[&dyn MirPass]` + optional `phase_change`, delegating to `run_passes_inner` (`:242`). Per pass: `should_run_pass` (`:214`, honoring `-Zmir-enable-passes`, gating, `is_enabled`); optional MIR dump before/after (`-Zdump-mir`, `:309`); run (profiled); optional validate/lint (`-Zvalidate-mir`/`-Zlint-mir`, `:334`). After the loop, a `phase_change` asserts the new phase is strictly later (`:345`), updates `body.phase`, resets `pass_count`. Phase ordering (`Analysis(Initial)` → `Analysis(PostCleanup)` → `Runtime(Initial)` → `Runtime(PostCleanup)` → `Runtime(Optimized)`) is enforced by the driver (`lib.rs:597`).

**Registration.** All MIR passes are enumerated by `declare_passes!` (`lib.rs:121`), building the static `PASS_NAMES` set (`lib.rs:94`); `run_passes_inner` cross-checks names (`:257`).

**Example.** `RemoveStorageMarkers` (`remove_storage_markers.rs:7`): `is_enabled` needs `mir_opt_level() > 0` (`:10`), `run_pass` strips `StorageLive`/`StorageDead`/`Nop` (`:14`), `is_required` is `true` (`:26`). Declared at `lib.rs:173`, slotted into `run_optimization_passes` at `lib.rs:739` (after inlining). That `run_optimization_passes` (`lib.rs:700`) is the canonical ordered list of ~40 passes — and it runs *inside* the `optimized_mir` query provider (`lib.rs:800`), so even the most pass-like part of rustc is reached lazily through a query. Other lists: `run_analysis_cleanup_passes` (`lib.rs:632`), `run_runtime_lowering_passes` (`lib.rs:651`, e.g. `ElaborateDrops`, `StateTransform`).

## Sense 2: the `rustc_passes` crate

A crate literally named "passes" — "Various checks" (`rustc_passes/src/lib.rs:1`). Modules: entry-point (`entry.rs`), lang items, reachability, dead-code, stability, etc. (`lib.rs:9`). Crucially **not** a fixed sequence — each is wired into the query system. `rustc_passes::provide` (`lib.rs:29`) registers each check as a query provider; e.g. `entry_fn` is a query (`entry.rs:23`), forced from the analysis driver via `tcx.ensure_ok().entry_fn(())` (`rustc_interface/src/passes.rs:1098`). So "passes" by name/spirit, dispatched on demand.

## Sense 3: lint passes

Lints are "passes" with their own traits (`rustc_lint/src/passes.rs`): `LateLintPass<'tcx>` (`:62`) and `EarlyLintPass` (`:181`), generated by macros so each is many empty-default `check_*` hooks. Many are fused into a single traversal via `declare_combined_late_lint_pass` (`:98`), producing e.g. `BuiltinCombinedModuleLateLintPass`. They run via `rustc_lint::check_crate`, called under the `lint_checking` timer (`rustc_interface/src/passes.rs:1238`).

## Contrast with the query system

None of these is a monolithic pipeline over the whole program. The MIR pass *lists* are real ordered sequences, but each executes inside a single query on one body, pulled lazily. The `rustc_passes` checks and lint passes are registered as / invoked through queries. The top-level analysis driver in `rustc_interface` is mostly `ensure_*` calls that *force* queries (`passes.rs:1090`), not a hand-rolled schedule.
