# How bootstrap works (stage0 → stage1 → stage2)

- **Upstream commit:** `4f8ea98e`
- **Area:** `src/bootstrap`, `src/stage0`
- **Date:** 2026-06-14

## Question

How does `bootstrap` work? rustc is written in Rust, so what compiles the
compiler in the first place?

## Short answer

It solves the chicken-and-egg problem with **stages**. `stage0` is a prebuilt
compiler **downloaded** from official servers (the previous `beta`). stage0
builds **stage1** from the current source; stage1 builds the stage1 std and then
the **stage2** compiler. stage2 is the fully self-hosted "real" compiler that
gets shipped. `bootstrap` (in `src/bootstrap`, written in Rust) is the build
system orchestrating all this, driven by the `x.py` entry point.

## Walkthrough

### The chicken-and-egg: stage0 is downloaded, not built

`upstream/src/stage0` records which prebuilt compiler to fetch:

```
dist_server=https://static.rust-lang.org
compiler_git_commit_hash=0417c25868d6dfbd1c291dfeae950504faa6f790
compiler_date=2026-05-31
compiler_version=beta
```

So **stage0 = a previous `beta` rustc + cargo + std, downloaded from
rust-lang.org**. This breaks the cycle. (The file is regenerated by
`./x.py run src/tools/bump-stage0`.)

### Build phases (from `src/bootstrap/README.md`)

1. **Entry point** `x.py` (= `bootstrap.py`; `x` on Unix, `x.ps1` on Windows)
   downloads the stage0 compiler/Cargo, **compiles the build system itself**
   (`src/bootstrap`, which is Rust) using stage0, then invokes that `bootstrap`
   binary.
2. The `bootstrap` binary reads config, runs sanity checks (toolchains, paths),
   and plans the build.
3. Staging:
   - **stage0 compiler** builds the **stage1 compiler** (linked against stage0's std).
   - **stage1 compiler** builds the **stage1 std**.
   - **stage1 compiler** builds the **stage2 compiler** (linked against stage1 std).

### Why go all the way to stage2

- **stage1** was produced by the *old* stage0 compiler, so it isn't identical to
  what the current source "should" produce.
- **stage2** is produced by a stage1 that was itself built from current source,
  i.e. **fully self-hosted** — this is the real, shippable compiler.
- Optionally a **stage3** can be built to check it matches stage2 (reproducibility).

Mental model: **borrow an old compiler (stage0) → build your own (stage1) →
use it to build the real one (stage2)**.

### Design goal

Per the README, each stage aims to (a) lean on **Cargo** as much as possible,
and failing that (b) lean on **Rust** as much as possible — i.e. keep the
Python (`bootstrap.py`) layer minimal; it's "only a symlink to bootstrap.py".
