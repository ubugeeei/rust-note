# Who manages Rust releases (the release process)

- **Upstream commit:** `4f8ea98e`
- **Area:** release process; `src/version`, `RELEASES.md`, `src/stage0`
- **Date:** 2026-06-15

## Question

Who is responsible for shipping Rust releases, and what is the pipeline moving a change from `nightly` to a stable release?

## Short answer

Rust's **Release team** owns the pipeline, operating a fixed **6-week train** across three channels (`nightly` → `beta` → `stable`). Every 6 weeks the current beta is promoted to stable, `main` is branched to a new beta, and nightly rolls on — which is why this checkout's `src/version` (`1.98.0`, nightly) is two cycles ahead of the newest stable in `RELEASES.md` (`1.96.0`). Stabilizing an individual feature is a separate, per-feature team process (FCP, flipping a feature gate), not the Release team's job. Critical fixes ship as **point releases** (e.g. `1.94.1`). **Editions** are an orthogonal opt-in mechanism, not tied to the cadence.

## The 6-week train model (evidence in this checkout)

- **nightly** — this branch. `src/ci/channel` = `nightly`; `src/stage0` has `nightly_branch=main`.
- **beta** — branched from main each release; becomes stable 6 weeks later.
- **stable** — the previous beta, promoted.

Concrete offset: `src/version` = `1.98.0`; latest stable in `RELEASES.md` = `1.96.0 (2026-05-28)`; `src/stage0` bootstrap = `compiler_version=beta`, `compiler_date=2026-05-31` (the 1.97 beta). nightly = stable + 2 (1.98 nightly, 1.97 beta, 1.96 stable) is exactly what the train predicts. `RELEASES.md` history shows the ~6-week cadence (1.96.0 2026-05-28, 1.95.0 2026-04-16, 1.94.0 2026-03-05, ...).

Promotion week (per forge.rust-lang.org, conventions can shift): bump the stable version number (Friday); Monday branch promotion (`start-release.py update-rust-branches`), plus a stable PR (sets `src/ci/channel` to `stable`) and a beta PR (sets it to `beta`); the new beta auto-releases at UTC midnight; the stable release publishes Thursday (~75–90 min).

## Version numbering and the files bumped

- **`src/version`** — single source of truth for the version string; bootstrap reads it (`src/bootstrap/src/lib.rs:517`; parsed as semver `config.rs:2208`).
- **`src/ci/channel`** — channel name (`nightly` here), flipped to `beta`/`stable` on those branches.
- **`RELEASES.md`** — human changelog; a new section per release/point release.
- **`src/stage0`** — pins the bootstrap compiler; its lower section is machine-generated by `./x.py run src/tools/bump-stage0`, listing `dist/<date>/...` artifacts + SHA-256 from `dist_server=https://static.rust-lang.org`. After a stable release, beta's stage0 is bumped to the new compiler. See [[04-bootstrap]].

## How stabilization works (separate from the train)

Per the rustc-dev-guide stabilization guide: write a stabilization report; **flip the feature gate** (move the entry from `compiler/rustc_feature/src/unstable.rs` to `accepted.rs`, remove gate-enforcement); ensure Reference docs; the owning team runs **FCP** (`@rfcbot fcp merge`). The feature lands on nightly and reaches users ~12 weeks later when the train carries it to stable. The Release team gates the *train*, not individual stabilizations.

## Release mechanics

- **Merge gate.** Changes land on `main` only via the merge queue / bors gate running the full CI; `src/stage0` shows `git_merge_commit_email=bors@rust-lang.org`.
- **Dist artifacts.** `dist/<date>/...tar.{gz,xz}`; servers in `src/stage0` (`dist_server`, `artifacts_server`). Published toolchains land on `static.rust-lang.org` with per-channel manifests.
- **Channels.** `src/ci/channel` determines version suffixes and feature-gate availability.

## Point releases

Out-of-band fixes ship as `X.Y.Z`: `RELEASES.md` shows `1.94.1 (2026-03-26)`, `1.93.1 (2026-02-12)`, etc. Backport workflow in `triagebot.toml`: `beta-nominated`/`beta-accepted`, `stable-nominated`/`stable-accepted` (`:26`, `:707`); nominating posts a Zulip poll (`/poll Approve stable backport of #...?` with an "approve (but does not justify new dot release on its own)" option, `:738`). Recent point releases correspond to security fixes (e.g. the CVE-2026-5222/5223 Cargo fixes under 1.96.0).

## Editions vs releases

A **release** (6-weekly) adds only backwards-compatible changes under the 1.0 guarantee. An **edition** (2015/2018/2021/2024) is the opt-in mechanism for otherwise-breaking changes; a newer compiler still compiles older editions and they interoperate. Rust 2024 shipped *with* 1.85.0 but editions aren't on the 6-week clock. See [[11-editions]].
