## Repository / build

| Term | Meaning |
|------|---------|
| `compiler/` | The rustc compiler itself â€” 75 `rustc_*` crates. See [[01-repo-layout]]. |
| `library/` | The standard library (`core`, `alloc`, `std`, ...). |
| `src/` | Everything *around* the compiler: build system, tools, docs, LLVM. **Not** the compiler. |
| `tests/` | The test suite (UI tests, mir-opt, run-make, ...). See [[02-scale-and-tests]]. |
| **bootstrap** | rustc's own build system (in `src/bootstrap`, written in Rust). Orchestrates the staged build. See [[04-bootstrap]]. |
| `x.py` / `x` / `x.ps1` | Entry-point wrapper that invokes bootstrap. `x` runs from any subdir. Ergonomic name, no official etymology. |
| `bootstrap.toml` | Build config file, generated by `./x.py setup`. |
| **profile** | A starting config preset chosen in `setup`: `compiler`, `library`, `tools`, `dist`. |
| **stage0** | A *prebuilt* compiler downloaded from rust-lang.org (previous `beta`). Breaks the chicken-and-egg cycle. |
| **stage1** | Compiler built from current source by stage0. |
| **stage2** | Compiler built by stage1 â€” the fully **self-hosted**, shippable rustc. |
| **self-hosting** | A compiler able to compile its own source. stage2 is the self-hosted result. |

## Compilation pipeline (IRs & phases)

| Term | Meaning |
|------|---------|
| **Token** | Output of the lexer (`rustc_lexer`); raw lexical units. |
| **AST** | *Abstract Syntax Tree* â€” direct parse of source (`rustc_ast`, produced by `rustc_parse`). |
| **HIR** | *High-level IR* â€” AST "lowered" to a more uniform, desugared tree (`rustc_hir`, via `rustc_ast_lowering`). Main IR for type checking. |
| **MIR** | *Mid-level IR* â€” a simplified control-flow-graph form used for borrow checking and optimization (`rustc_mir_build`, `rustc_mir_transform`). |
| **codegen** | Turning MIR into machine code, usually via the LLVM backend (`rustc_codegen_ssa` + `rustc_codegen_llvm`). |
| **lowering** | Translating one IR into a lower-level one (ASTâ†’HIR, HIRâ†’MIR). |
| **desugaring** | Rewriting convenient syntax into a smaller core form (e.g. `for` loops â†’ loop+iterator) during lowering. |
| **name resolution** | Binding identifiers to their definitions (`rustc_resolve`). |
| **type inference** | Deducing unannotated types (`rustc_infer`, `rustc_hir_typeck`). |
| **borrow check** | Verifying ownership/lifetime rules on MIR (`rustc_borrowck`). |
| **monomorphization** | Generating concrete copies of generic code per type instantiation (`rustc_monomorphize`). |

## Compiler architecture

| Term | Meaning |
|------|---------|
| **query system** | rustc's demand-driven model: results are requested as "queries", computed lazily, and cached (enables incremental compilation). See [[06-reading-the-compiler]]. |
| **`TyCtxt`** | The central typing context in `rustc_middle` â€” the hub through which queries and type info are accessed. |
| `rustc_middle` | The "middle" crate holding shared core data (incl. `TyCtxt`, `ty`); common dependency of most passes. |
| `rustc_driver` / `rustc_driver_impl` | The CLI entry/driver that starts a compile. |
| `rustc_interface` | Drives the phases in order; `passes.rs` is the phase "table of contents". |
| **interning** | Storing one canonical copy of a value (e.g. `Symbol`, `Ty`) so they compare by pointer/id. |
| **arena** | Bump allocator that owns many objects with a shared lifetime (`rustc_arena`). |
| **LLVM** | The default codegen backend (`src/llvm-project` submodule). Alternatives: Cranelift, GCC. |

## Testing

| Term | Meaning |
|------|---------|
| **UI test** | A small program whose compiler output (errors/warnings) is diffed against a checked-in `.stderr`/`.stdout`. Largest test category. |
| **compiletest** | The test runner that executes the `tests/` suites (`src/tools/compiletest`). |
| **revisions** | Multiple sub-test configurations declared inside a single test file. |
