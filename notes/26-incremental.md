# Incremental compilation (`rustc_incremental` + the dep graph)

- **Upstream commit:** `4f8ea98e`
- **Area:** `compiler/rustc_incremental`, `compiler/rustc_middle/src/dep_graph`, `compiler/rustc_query_impl`
- **Date:** 2026-06-15

> Note: at this commit there is **no `compiler/rustc_query_system` crate**. The red-green engine, color map, `try_mark_green`, and dep-graph serialization live in `compiler/rustc_middle/src/dep_graph/{graph.rs,dep_node.rs,serialized.rs}`; the per-query execution driver is in `compiler/rustc_query_impl/src/execution.rs`. `rustc_incremental` is the persistence layer.

## Question

How does rustc reuse work between compilations — the dep graph (DepNode/DepGraph), the red-green algorithm, on-disk caching, and the split between `rustc_incremental` and the query engine?

## Short answer

Every tracked query invocation is a `DepNode` (a `DepKind` + a 128-bit key fingerprint) in a dependency graph; edges record which queries read which others (`dep_graph/dep_node.rs:93`, `graph.rs:460`). At session start rustc loads the *previous* session's serialized graph and an on-disk result cache, then lazily re-validates: `try_mark_green` walks a node's previous-session dependencies, and a node becomes **green** (reusable) only if all deps are green, otherwise rustc forces the query and compares result fingerprints to decide green-or-**red** (`graph.rs:861`, `:744`). The colored-marking/hashing/serialization engine lives in `rustc_middle/src/dep_graph` + `rustc_query_impl`; **`rustc_incremental` is almost entirely persistence** — the incr-comp session directory, loading/saving `dep-graph.bin` and `query-cache.bin`, work products, and GC (`rustc_incremental/src/persist/{fs,save,load}.rs`).

## The DepNode and the DepGraph

```rust
pub struct DepNode { pub kind: DepKind, pub key_fingerprint: PackedFingerprint }
```
(`dep_node.rs:93`). The `key_fingerprint` is a session-independent stable hash of the query key — like a git hash, it identifies the same logical node across sessions. `DepKind` is generated by `define_dep_nodes!`, one variant per query plus specials (`DepKind::Red` `graph.rs:159`, `DepKind::SideEffect` `:691`).

The live graph is `DepGraphData` (`graph.rs:99`), holding: `previous: Arc<SerializedDepGraph>` (last session, `:108`), `current: CurrentDepGraph` (this session, streamed straight to disk, never removed, `:104`), and `colors: DepNodeColorMap` (`:110`). A running query's `with_task` wraps the provider in a `TaskDeps` that collects every node read via `read_index`, then interns a node with those edges (`:325`, `:460`). `eval_always` queries skip tracking and depend on `FOREVER_RED_NODE`, so they always re-run (`:344`).

## Fingerprints

Two distinct ones: the **key fingerprint** (which node), and the **value fingerprint** (`hash_result`, `:124`) — a stable hash of the result. The value fingerprint decides green vs red when a query is re-executed: `alloc_and_color_node` compares the fresh `value_fingerprint` against `previous.value_fingerprint_for_index(...)` — equal → green, different/absent → red (`:744`).

## The red-green algorithm (`try_mark_green`, `graph.rs:861`)

For a `DepNode`:
1. Not in previous session → `None` (must run).
2. Already colored: `Green` → reuse; `Red` → `None`; `Unknown` → try to prove green.
3. `try_mark_previous_green` iterates the node's *previous* dependency edges (`:889`): dep `Green` → continue; dep `Red` → bail (`:908`); dep `Unknown` → recurse, and if that fails, **force** the dep (re-execute) and re-check (`:914`).
4. If all deps ended green, promote the node and color it green (`:957`).

Colors are stored as one `AtomicU32` per previous node with sentinels (`COMPRESSED_RED`/`COMPRESSED_UNKNOWN`), making coloring lock-free and parallel-safe (`:1263`). The query layer drives this in `rustc_query_impl/src/execution.rs`: `execute_job_incr` tries `try_mark_green`; on success `load_from_disk_or_invoke_provider_green` loads the cached result or recomputes (`:440`). `incremental_verify_ich` re-hashes sampled disk-loaded results to catch unstable query impls (`:528`).

## On-disk cache and the session directory

`rustc_incremental` owns the layout (`persist/fs.rs`). Per crate, an incr-comp dir holds per-session dirs named `s-{timestamp}-{random}-working` while in progress; on clean finish `-working` is replaced by the crate's SVH (`fs.rs:14`, `finalize_session_directory` `:307`). Three files (`fs.rs:128`): `dep-graph.bin` (the `SerializedDepGraph`), `query-cache.bin` (the `OnDiskCache`), `work-products.bin` (object files etc. keyed by `DepNode::CodegenUnit`).

- **Load** (`persist/load.rs`): copy/hard-link the previous finalized dir into a fresh `-working` dir, then `load_dep_graph` (checks the command-line-args hash; mismatch purges, `:180`); reload work products (drop any with missing files); load the query cache lazily (`:127`).
- **Save** (`persist/save.rs`): rename the staging file to `dep-graph.bin` and serialize the `OnDiskCache`, only if no errors (`:25`). `exec_cache_promotions` pulls green-but-undecoded values into memory so they get re-written (`graph.rs:998`).
- **GC**: `garbage_collect_session_directories` deletes abandoned `-working` dirs (failed `flock`) and keeps only the most recent finalized dir per crate (`fs.rs:42`).

## Worked example: edit one fn body

Change only `fn foo`'s body and rebuild:
1. The file hash changes → HIR nodes covering `foo`'s body are forced; value fingerprints differ → those nodes go **red**.
2. `typeck(foo)` depends on `foo`'s body HIR. `try_mark_green(typeck(foo))` hits a red dep and bails (`:908`); it re-executes. Same result hash → green-after-recompute; different → red.
3. `typeck(bar)` (unrelated) has no edge to `foo`'s body; all deps green → `try_mark_green` succeeds and the result is loaded from `query-cache.bin` — no recompute.
4. `optimized_mir(foo)` was red → its CGU's `WorkProductId` is dirty, the old `.o` is discarded and regenerated; CGUs with only unchanged items reuse their saved object files.

So an edit's "red wave" propagates only along real read-edges; fingerprint-equal queries short-circuit it (the green firewall).
