# SIMD in Rust

- **Upstream commit:** `4f8ea98e`
- **Area:** `library/core` (arch/simd), stdarch/portable-simd, `compiler/rustc_codegen_*`
- **Date:** 2026-06-15

## Question

How does Rust expose SIMD across its layers ‚Äî vendor intrinsics (`core::arch`), portable `core::simd`, the compiler's `repr(simd)`/`simd_*` machinery, and LLVM autovectorization ‚Äî and where does each live?

## Short answer

Two user-facing surfaces: **target-specific vendor intrinsics** under `core::arch::{x86_64, aarch64, ...}` (from the **stdarch** subtree, with runtime detection like `is_x86_feature_detected!`), and **portable SIMD** under the nightly `core::simd` (`Simd<T, N>` from the **portable-simd** subtree). Both bottom out in the same primitives: `#[repr(simd)]` vector types + `#[rustc_intrinsic]` `simd_*` functions in `library/core/src/intrinsics/simd/mod.rs`. The LLVM backend lowers those to native vector ops in `rustc_codegen_llvm/src/intrinsic.rs::generic_simd_intrinsic`. **Autovectorization** is a separate implicit path: LLVM's loop/SLP vectorizers turn scalar loops into vector code, gated on opt level. (In this tree, stdarch and portable-simd are vendored **subtrees**, present and readable, not `.gitmodules` entries.)

## Layer 1 ‚Äî Vendor intrinsics (`core::arch`)

`core` pulls in stdarch's `core_arch` by path:

- `library/core/src/lib.rs:363` ‚Äî `#[path = "../../stdarch/crates/core_arch/src/mod.rs"] mod core_arch;`
- `library/core/src/arch.rs:13` ‚Äî `pub use crate::core_arch::arch::*;` (stable since 1.27).

Per-arch intrinsics live under `library/stdarch/crates/core_arch/src/{x86, aarch64, ...}/`. Example (`x86/sse2.rs:104`):

```rust
#[target_feature(enable = "sse2")]
#[stable(feature = "simd_x86", since = "1.27.0")]
pub const fn _mm_add_epi32(a: __m128i, b: __m128i) -> __m128i {
    unsafe { transmute(simd_add(a.as_i32x4(), b.as_i32x4())) }
}
```

Note the vendor intrinsic is itself implemented over the portable `simd_add` compiler intrinsic (Layer 3). `#[target_feature(enable="sse2")]` makes it unsafe to call unless statically enabled or runtime-guarded. **Runtime detection** comes from std_detect: `is_x86_feature_detected!` is generated by the `features!` macro (`library/std_detect/src/detect/arch/x86.rs:18`), reading CPUID once into a global and checking a bit per feature.

## Layer 2 ‚Äî Portable SIMD (`core::simd` / `std::simd`)

Pulled into `core` the same way (`library/core/src/lib.rs:386`), gated behind `portable_simd` (nightly, tracking issue 86656). Central type (`library/portable-simd/crates/core_simd/src/vector.rs:106`):

```rust
#[repr(simd, packed)]
pub struct Simd<T, const N: usize>([T; N]) where T: SimdElement;
```

So `Simd<T, N>` is just a `#[repr(simd)]` newtype over `[T; N]`; its ops call the `simd_*` intrinsics ‚Äî portable SIMD and vendor intrinsics share the same lowering.

## Layer 3 ‚Äî Compiler support: `repr(simd)` + `simd_*`

**The attribute.** `#[repr(simd)]` sets a flag in `rustc_abi/src/lib.rs:84` (`IS_SIMD`), queried via `is_simd()` (`:167`). Layout/validation in `rustc_abi/src/layout.rs` (`simd_type` `:223`; errors `ZeroLengthSimdType`/`OversizedSimdType`/`NonPrimitiveSimdType`) ‚Äî a SIMD type must be a non-empty, non-oversized vector of primitive lanes.

**The intrinsics.** Generic `simd_*` ops are declared `#[rustc_intrinsic]` (no bodies) in `library/core/src/intrinsics/simd/mod.rs`: `simd_insert`/`simd_extract` (`:18`/`:29`), `simd_add`/`sub`/`mul`/`div`/`rem` (`:70`+), comparisons (`:288`+), `simd_shuffle` (`:359`), `simd_gather`/`scatter` (`:380`/`:404`), `simd_reduce_*` (`:488`+). The doc: *"a 'vector' is any `repr(simd)` type"* (`:3`).

**Lowering to LLVM.** `rustc_codegen_llvm/src/intrinsic.rs`: `codegen_intrinsic_call` (`:174`) dispatches SIMD names to `generic_simd_intrinsic` (`:1944`). Arithmetic is lowered by the `arith_binary!` macro (`:3231`):

```rust
arith_binary! {
    simd_add: Uint, Int => add, Float => fadd;
    simd_mul: Uint, Int => mul, Float => fmul;
    // ...
}
```

So `simd_add` on `i32x4` becomes a single LLVM `add <4 x i32>` ‚Äî no loop. `simd_shuffle` is special-cased (`:2211`). Other backends mirror this (`rustc_codegen_cranelift/src/intrinsics/simd.rs`, `rustc_codegen_gcc/src/intrinsic/simd.rs`); Miri handles them in `rustc_const_eval/src/interpret/intrinsics/simd.rs`. Misuse ‚Üí `E0511`/`E0439`.

## Layer 4 ‚Äî Autovectorization (implicit)

Distinct from explicit SIMD: ordinary scalar loops turned into vector code by LLVM's Loop and SLP vectorizers, controlled purely by opt level (`rustc_codegen_ssa/src/back/write.rs:228`):

```rust
vectorize_loop: !no_vectorize_loops && (optimize == More /*-O2*/ || Aggressive /*-O3*/),
vectorize_slp:  !no_vectorize_slp  &&  optimize == Aggressive /*-O3*/,
```

So loop-vectorize at `-O2`/`-O3`, SLP only at `-O3`; disable via `-Cno-vectorize-loops`/`-Cno-vectorize-slp`. Threaded into the LLVM pass-manager run (`rustc_codegen_llvm/src/back/write.rs:780`). No `unsafe`, no nightly, no special types ‚Äî but best-effort, not guaranteed (the tradeoff vs explicit SIMD). See [[12-optimization-pipeline]].

## Small example

```rust
// (A) Explicit portable SIMD -> simd_add -> one LLVM `add <4 x i32>`
#![feature(portable_simd)]
use std::simd::Simd;
fn add4(a: [i32;4], b: [i32;4]) -> [i32;4] {
    (Simd::from_array(a) + Simd::from_array(b)).to_array()
}

// (B) Scalar loop -> Layer 4 may vectorize at -O2/-O3 (no nightly, no unsafe)
fn add_scalar(a: &[i32], b: &[i32], out: &mut [i32]) {
    for i in 0..out.len() { out[i] = a[i] + b[i]; }
}
```

(A) is deterministic vector codegen via the intrinsic path; (B) the optimizer *may* vectorize depending on opt level and aliasing/length analysis. The vendor version (Layer 1) is `_mm_add_epi32` over `__m128i` behind a `target_feature`/runtime guard ‚Äî itself wrapping `simd_add`.

## Notes

- stdarch/portable-simd are **subtrees** here, not submodules (`.gitmodules` doesn't list them).
- There's also a scalable-vector path (`library/core/src/intrinsics/simd/scalable.rs`, `SimdVectorKind::Scalable`) for SVE-style targets ‚Äî noted, not deep-dived.
