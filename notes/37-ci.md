# rustc CI: jobs, runners, and timing

- **Upstream commit:** `4f8ea98e`
- **Area:** `.github/workflows`, `src/ci`
- **Date:** 2026-06-15

## Question

How is rustc's CI structured: what jobs run, on which runners, driven by what, and roughly how long does a full run take?

## Short answer

rustc's CI is GitHub Actions, but the job matrix is *generated*, not hand-written: a `calculate_matrix` job reads `src/ci/github-actions/jobs.yml` via the `citool` crate and emits the matrix `ci.yml` consumes. Jobs are grouped by trigger into `pr` (11 jobs), `try` (1), `optional`, and `auto` (83 jobs in this checkout). The full `auto` matrix runs through bors on `automation/bors/auto` and must be green before a commit lands. Linux jobs run in Docker images under `src/ci/docker/`; runners range from free 4-core GitHub runners (`ubuntu-24.04`) up to large/custom runners (8/16-core paid + AWS CodeBuild 36-core). Each job is capped at `timeout-minutes: 360`; a full `auto` run is dozens of jobs in parallel, roughly a few hours wall-clock.

## How CI is wired

`.github/workflows/ci.yml` header: "uses job definition data from src/ci/github-actions/jobs.yml. You should primarily modify the `jobs.yml` file." Flow:
1. `ci.yml` triggers on `pull_request` and push to `automation/bors/auto`, `automation/bors/try`, `try-perf` (`ci.yml:11`).
2. `calculate_matrix` (on `ubuntu-24.04-arm`) invokes `citool` to compute the matrix, exposing `run_type`/`jobs` (`ci.yml:47`).
3. The `job` matrix is `include: ${{ fromJSON(needs.calculate_matrix.outputs.jobs) }}`, `runs-on: ${{ matrix.os }}`, `timeout-minutes: 360` (`ci.yml:72`).

`citool` maps trigger → run type (`src/ci/citool/src/main.rs:43`): push to `automation/bors/auto` → `AutoJob`; try → `TryJob`; else PR set. Invariant (`citool/src/jobs.rs:117`): every PR job is a subset of Auto jobs.

## Job categories (from `jobs.yml`)

- **`pr`** (11): every PR push; `PR_CI_JOB: 1`. E.g. `pr-check-1/2`, `tidy` (sets `SKIP_SUBMODULES: src/gcc`), `x86_64-gnu-llvm-21`, `aarch64-gnu-llvm-21-1/2`, `x86_64-gnu-tools`, `x86_64-gnu-miri`, `x86_64-gnu-gcc`.
- **`try`** (1): `@bors try` builds — `dist-x86_64-linux-quick` (alias of `dist-x86_64-linux`); `DIST_TRY_BUILD` skips components/tests.
- **`optional`**: only on request (`@bors try jobs=<name>` / `try-job:`). Currently `optional-pr-check-1`.
- **`auto`** (83): full merge matrix; must be green to merge; "Auto jobs may not specify `continue_on_error: true`" (`jobs.yml:191`).

## The matrix (real job names)

- **Linux/Docker** (the bulk): `dist-*` cross builds (`dist-aarch64-linux`, `dist-android`, `dist-riscv64-linux`, `dist-s390x-linux`, `dist-various-1/2`, `dist-x86_64-linux`, `dist-x86_64-musl`, ...); test builders (`aarch64-gnu`, `i686-gnu-1/2`, `test-various`, `x86_64-gnu`, `x86_64-gnu-tools`, `x86_64-gnu-miri`, `x86_64-rust-for-linux`); heavy jobs split N-ways via `DOCKER_SCRIPT` (`x86_64-gnu-llvm-21-{1,2,3}`, `-22-{1,2,3}`).
- **macOS**: `dist-x86_64-apple`, `dist-apple-various`, `dist-aarch64-apple`, `aarch64-apple`, experimental `aarch64-apple-macos-26`.
- **Windows**: MSVC split `x86_64-msvc-1/2`, `i686-msvc-1/2`, `aarch64-msvc-1/2`, ext jobs, MinGW, and dists.

## Runners (`runners:` anchors, `jobs.yml:3`)

| Anchor | `os` label | Class |
|---|---|---|
| `job-linux-4c` | `ubuntu-24.04` | free runner (`free_disk: true`) |
| `job-linux-8c` | `ubuntu-24.04-8core-32gb` | large |
| `job-linux-16c` | `ubuntu-24.04-16core-64gb` | large |
| `job-aarch64-linux` | `ubuntu-24.04-arm` | free arm |
| `job-macos-15` | `macos-15` (Arm64) | macOS |
| `job-windows` | `windows-2025` | Windows |
| `job-linux-36c-codebuild` | `codebuild-ubuntu-22-36c-...` | custom AWS CodeBuild |

CodeBuild runners are provisioned in `rust-lang/simpleinfra`. The heaviest dist job, `dist-x86_64-linux` (`CODEGEN_BACKENDS: llvm,cranelift`, `DOCKER_SCRIPT: dist.sh`), uses the 36-core CodeBuild runner; most Linux jobs use the free 4-core runner with `free_disk: true` (`ci.yml:129` calls `free-disk-space-linux.sh`).

## Docker-based Linux jobs (`src/ci/docker/`)

`src/ci/docker/run.sh` builds/runs the image named by `IMAGE` (default = job name). Images live under `src/ci/docker/host-x86_64/` (~56) and `host-aarch64/`. One image can serve multiple jobs (e.g. `x86_64-gnu-llvm-21-*` share an image, varying `DOCKER_SCRIPT`). Locally: `cargo run --manifest-path src/ci/citool/Cargo.toml run-local <job>`.

## Timing

Repo-grounded: each job capped at `timeout-minutes: 360` (a ceiling, not typical). The repo doesn't state aggregate wall-clock. Approximate (web context, not verifiable here): PR CI ~1h; a full Auto run ~3h wall-clock with dozens of jobs in parallel; historically (~2023) ~400 CPU-hours across 60+ jobs — treat as approximate (the matrix is now 83 auto jobs).
